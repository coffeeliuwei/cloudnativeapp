## Part 1 信息安全等级保护（等保 2.0）

> 一句话：**等保 = 中国法律强制要求的"安全合规体检"**。所有面向中国用户提供互联网服务的系统，超过一定规模必须过等保。不是技术问题，是法律问题。

### 1.1 等保 5 个级别

| 级别 | 一般对应什么系统 |
|------|---------------|
| 一级 | 个人博客、小作坊网站（基本不强制） |
| **二级** | **大多数中小企业互联网应用** |
| **三级** | **重要业务系统**（电商、金融、医疗、教育、地市级政务） |
| 四级 | 国家级核心系统（银行总行、央企核心） |
| 五级 | 国防、绝密（基本看不到） |

**最常见是二级 / 三级**——本课程 coffee 项目这种规模上线了对外服务，按规定就要走 **等保二级**；如果接了用户支付变商业应用，要 **等保三级**。

### 1.2 等保具体查什么

不是查"有没有"，而是查 **"做没做到"**。10 个大类（简略版）：

- 物理与环境（机房）
- 网络与通信（防火墙、VPN）
- 设备与计算（主机加固、堡垒机）
- 应用与数据（WAF、加密、备份）
- 安全管理制度（文档）
- 安全管理机构（专人负责）
- 安全管理人员（培训）
- 安全建设管理（开发规范）
- 安全运维管理（监控、应急）
- 数据安全（个保法、数据分类）

**好消息**：上阿里云做等保，**前 4 类基本由阿里云的产品满足**（云防火墙、堡垒机、WAF、KMS 等本章后续的产品），**后 6 类是你自己的管理制度**——写文档、定流程、留记录。

### 1.3 本课程项目能不能过等保

按现在的样子 **过不了**——RDS 公网白名单全开、ECS root 直登、没 WAF……
**按本章 Part 4-8 全配齐后 → 二级基本能过；三级需要再补一些定期渗透测试和管理文档**。

## Part 2 上云安全的基础——共享责任模型

> 不搞清这个，所有后续工作要么白做要么做漏。**"上云后阿里云帮我搞定一切安全"——这是新手最大的误区**。

### 2.1 一张图说清楚

```
            ┌──────────────────────────────────┐
            │   你的应用代码 + 业务逻辑          │
            │   你的数据 + 数据加密策略          │
你的责任  │   ────────────────────────────  │
            │   操作系统补丁（ECS 上）           │
            │   账号权限（RAM 配置）            │
            │   网络访问控制（安全组、WAF 规则） │
            ├──────────────────────────────────┤
阿里云责任│   虚拟化层、宿主机、数据中心物理安全 │
            │   电力、网络骨干、机房消防          │
            └──────────────────────────────────┘
```

### 2.2 什么 IaaS / PaaS / Serverless 边界不一样

| 服务形态 | 谁打操作系统补丁 | 谁配运行时 | 谁管中间件版本 |
|---------|---------------|----------|--------------|
| **ECS（IaaS）** | **你** | **你** | **你** |
| **EDAS（PaaS）** | 阿里云 | 阿里云 | **你** |
| **SAE（Serverless）** | 阿里云 | 阿里云 | 阿里云（部分） |

> 越靠 PaaS / Serverless，阿里云替你承担的安全责任越多——这是选 SAE 不只是"省运维"，**安全责任也"省一截"** 的另一个理由（呼应第 11 章 Part 7）。

### 2.3 本项目落实共享责任的 5 件事

1. **你这边的代码 / 配置**：永远 `git` 留痕，密码 / 密钥不进代码（用环境变量或 KMS）
2. **ECS 操作系统**：开 **自动安全补丁**（阿里云控制台一键开）
3. **账号**：废 Root，全部用 RAM 子账号（第 13 章 Part 5 已讲）
4. **网络**：所有公网入口前面挂 WAF + 安全组最小授权
5. **数据**：敏感字段加密（Part 7 详讲）

## Part 3 漏洞扫描与攻防演练

> 一句话：**"先自己打自己一遍，找出弱点早修——比等真黑客来找好得多"**。

### 3.1 三种"自己打自己"的方式

| 方式 | 谁来打 | 适合 |
|------|------|------|
| **自动漏洞扫描** | 工具（云安全中心 / 第三方扫描器） | 已知漏洞、配置错误 |
| **众测 / 漏洞赏金** | 阿里云 **"先知"** 平台上的白帽黑客 | 业务逻辑漏洞、未知 0day |
| **红蓝对抗演练** | 内部安全团队 vs 业务团队 | 大公司、护网行动前 |

### 3.2 阿里云"先知"众测平台

**先知**（Mingyan）是阿里云运营的众测平台——**你出赏金、白帽黑客来打你**。

模式：
1. 你在先知发布"招募贴"：告诉白帽你哪些资产可测、奖金多少
2. 白帽黑客来攻击你的系统，发现漏洞提交报告
3. 漏洞复现确认后，按严重程度发奖金（几百 ~ 几万）

**为什么这是个好生意**：
- 一次内部渗透测试动辄几十万、几个白帽
- 先知模式按结果付费，**没漏洞不花钱**
- 白帽人数比你内部团队多 10 倍以上

> 📷 截图占位：先知平台首页
> 🔗 官方文档：[阿里云先知（安全众测）](https://help.aliyun.com/zh/security-vulnerability-management)

### 3.3 给本项目做漏洞扫描（最小操作）

1. 云安全中心控制台 → **漏洞管理 → 立即检测**
2. 扫描会发现常见类型：
   - 操作系统漏洞（OpenSSL CVE、glibc CVE）
   - Web 应用漏洞（弱口令、目录遍历）
   - 应用基线（SSH 配置 / MySQL 配置）
   - 应急漏洞（log4j2 这种）
3. **按 CVSS 评分排优先级修**：紧急 > 高危 > 中危
4. 修完再扫一次

## Part 4 云安全中心 SAS — 主机层 EDR

> **对照前面章节**：你 06 章建了 3 台 ECS——**有没有人在你 ECS 上挖矿你能知道吗？有没有 Web Shell 被植入你能知道吗？** 云安全中心装个 Agent，这些都告诉你。

### 4.1 云安全中心解决什么

云安全中心 = **EDR**（端点检测与响应，Endpoint Detection & Response）+ **漏洞 / 基线 / 入侵检测**的一体化产品。

5 大核心能力：

| 能力 | 干什么 |
|------|--------|
| **资产梳理** | 自动盘点云上所有 ECS / RDS / OSS 等，谁是谁的责任人 |
| **风险发现** | 操作系统漏洞 + 应用漏洞 + 基线不合规 + AK 泄露 |
| **安全加固** | 给出修复建议或一键修复 |
| **实时防护** | 病毒 / 木马 / 异常登录 / Web Shell 实时拦截 |
| **检测响应** | 入侵告警 + 取证 + 隔离 |

**核心机制**：在每台 ECS 装一个 **云安全中心 Agent**（Linux/Windows 都支持），Agent 把进程、文件、网络连接、登录事件上报到云端，云端 AI 分析后给告警。

### 4.2 接入本项目的 3 台 ECS（5 分钟）

1. 云安全中心控制台 → **资产管理 → 资产中心**
2. 阿里云 ECS 通常已自动接入，看是否在线
3. 若离线，按提示在 ECS 上跑一行命令装 Agent：
   ```bash
   wget http://update2.aegis.aliyun.com/download/install.sh && \
   chmod +x install.sh && ./install.sh
   ```
4. 等 1 分钟，控制台显示"在线"

   > 📷 截图占位：云安全中心资产中心页面
   > 🔗 官方截图参考：[云安全中心产品概述](https://help.aliyun.com/zh/security-center/product-overview/what-is-security-center)

### 4.3 装上 Agent 后能看到的"魔法"

举几个真实场景（云安全中心日常会喊的事）：

- "ECS-1 检测到挖矿木马连接 *.xmr.org，已隔离"
- "ECS-2 检测到 root 用户在 03:47 从俄罗斯 IP 登录成功，疑似异常"
- "ECS-3 上传了一个 Webshell 文件 `/tmp/x.jsp`，已删除"
- "你的 AccessKey AKIA*** 在 GitHub 公共代码里被检索到，**立刻轮换**"

**最后这一条是新手最常踩的坑** —— 把含 AK 的配置文件 push 到 GitHub。云安全中心 **每天扫 GitHub 找你账号下泄露的 AK**——这是个救命功能。

## Part 5 网络防护三板斧 — 云防火墙 / WAF / DDoS

> 三个产品都是流量层防御，**但解决不同的事**：

| 产品 | 防什么 | 在 OSI 哪一层 |
|------|--------|-------------|
| **DDoS 防护** | 大流量打死你 | L3/L4 网络层 + L7（高防能防 CC） |
| **云防火墙** | "谁能访问谁" 这件事 | L3/L4 网络层 |
| **WAF** | SQL 注入 / XSS / Webshell 上传 | L7 应用层 |

### 5.1 DDoS 防护 3 个层级

| 产品 | 价格 | 防什么 | 接入 |
|------|------|------|------|
| **DDoS 基础防护** | **免费** | 500M ~ 5G 流量 | 阿里云产品默认开 |
| **DDoS 原生防护** | 付费 | 大流量，**旁路清洗** | 加在阿里云资源上 |
| **DDoS 高防（含国际高防）** | 付费 | **超大流量 + CC 攻击 + 可保非阿里云资源** | **代理清洗模式**，把流量引到高防再回源 |

**怎么选**：
- 教学项目流量小 → **DDoS 基础防护** 就够了，免费
- 上线后偶有小流量攻击 → **DDoS 原生防护**
- 互联网业务，可能被打 100G+ → **DDoS 高防**（年费几万起）
- 业务在多家云 / 本地 IDC → **DDoS 高防**（只有它支持非阿里云源站）

### 5.2 Web 应用防火墙 WAF

WAF 防的是 **应用层攻击**：

- **OWASP Top 10**：SQL 注入、XSS、命令注入、文件上传、SSRF 等
- **CC 攻击**：高频请求耗资源（"假装合法用户的 DDoS"）
- **爬虫与 BOT**：恶意爬虫、刷优惠券
- **0day 漏洞**：log4j2 这种紧急虚拟补丁

**接入方式 2 选 1**：

| 接入方式 | 怎么生效 | 适合 |
|---------|---------|------|
| **CNAME 接入** | 把域名 CNAME 到 WAF，流量先过 WAF 再回源 | 通用，灵活，支持源站在任何地方 |
| **透明接入** | 给阿里云 ECS / SLB 一键开 WAF，无需改 DNS | 源站在阿里云、不想动 DNS |

**给本项目接 WAF（CNAME 方式）**：
1. WAF 控制台 → 添加防护对象 → 域名 `coffee.example.com`
2. 控制台返回一个 WAF CNAME 地址 `*.alicdn.com`
3. 回云解析 DNS → 把 `coffee` 主机记录的解析值改成这个 CNAME
4. **流量瞬间过 WAF**，恶意请求被拦截

   > 📷 截图占位：WAF 控制台 + 接入向导
   > 🔗 官方文档：[WAF 产品概述](https://www.aliyun.com/product/waf)

### 5.3 云防火墙

云防火墙做的是 **"网络访问控制"**——具体管 3 类边界：

| 边界 | 管什么流量 | 替代了什么 |
|------|---------|-----------|
| **互联网边界** | 公网 ↔ 你的资源（南北向） | 比"安全组"更集中可视化 |
| **VPC 边界** | VPC ↔ VPC | 多 VPC 互访的统一管控 |
| **主机边界** | ECS ↔ ECS（东西向，**微隔离**） | 防止"一台被攻破，横向移动" |

**微隔离的意义**：传统架构里一台 Web 机器被攻破后，黑客可以从这台机器横向跳到数据库机器。**云防火墙的主机边界策略让"Web 机器只能访问数据库的 3306 端口，不能访问任何别的端口"**——把横向移动堵死。

> 教学项目可以先不开云防火墙（用安全组也能管基础访问控制），上线后业务规模上来再加。

## Part 6 应用安全防护 — 代码层纵深防御

> **网络层防住了 99%，剩下 1% 会用合法请求漏过 WAF**——这时候要在应用代码层再来一道防御。

### 6.1 RASP 是什么

**RASP（Runtime Application Self-Protection，运行时应用自我保护）** 装在 Java 应用进程里（类似 ARMS Agent 的形态），**在代码执行的瞬间** 检查：

- "这个 SQL 语句是不是被注入了？"
- "这次 `Runtime.exec()` 是不是被恶意构造的？"
- "这次反序列化是不是要触发 RCE？"

**RASP vs WAF**：

| 维度 | WAF | RASP |
|------|-----|------|
| 工作位置 | 流量入口（看请求） | **应用进程内**（看代码执行） |
| 能防的最高境界 | 已知攻击模式 | **零日漏洞**（哪怕 WAF 漏了，代码层还能挡） |
| 误报率 | 较高 | 较低 |
| 性能 | 几乎零开销 | < 3% 性能损耗 |

**阿里云云安全中心高级版 / 企业版** 内置 RASP 能力（Java / PHP / Node.js），开关一键打开。

### 6.2 API 安全

云原生应用对外暴露的几乎全是 API（REST / GraphQL / gRPC）。新一代攻击越来越聚焦 API：

- **数据爬取**：合法的 API 请求被刷取了百万条
- **越权访问**：用 A 用户 token 查 B 用户数据
- **批量遍历**：枚举手机号查会员信息

应对方式：
- **API 网关 + 限流**（接到本项目 coffee-app 网关层）
- **WAF 的 API 安全模块**（新一代 WAF 已开始覆盖）
- **行为分析**（看用户访问模式异常）

### 6.3 给本项目加一道应用层防护

最务实的 3 件事：
1. **coffee-app 网关层加限流**（Sentinel / Resilience4j）
2. **关键接口加签名校验**（防 API 被刷）
3. **云安全中心打开 RASP**（如果用了企业版）

## Part 7 数据安全 — 最后一道墙

> **前面所有防御都被攻破后，黑客拖走的数据库——如果是加密的就是一堆乱码；如果明文，就是事故新闻头条**。这是为什么数据安全被称为"核心"和"最后一道墙"。

### 7.1 数据安全 3 层防护

| 层级 | 阿里云对应产品 | 做什么 |
|------|------------|--------|
| **存储加密** | RDS / OSS / NAS 透明加密 | 磁盘级加密，数据落盘就是密文 |
| **传输加密** | HTTPS / TLS | 数据在网络上传输是密文 |
| **应用层加密** | **KMS**（密钥管理服务） | 你应用层主动加密敏感字段 |

### 7.2 KMS（密钥管理服务）

**KMS 的核心价值**：你不再需要自己保管加密密钥——密钥放 KMS，应用调 KMS API 加解密。**KMS 用阿里云硬件加密机（HSM）保管密钥**，连阿里云自己工程师都看不到。

典型用法：

```
应用层敏感字段（手机号 / 身份证）
        │
        │ 1. 调 KMS Encrypt API（输入明文，输出密文）
        ▼
   存进 RDS（密文）
        │
        │ 2. 查询时从 RDS 取出密文
        │ 3. 调 KMS Decrypt API（输入密文，输出明文）
        ▼
   返回给前端
```

**为什么不用自己写 AES 加密**：密钥怎么保存是个深坑——写代码里？写配置文件？写另一张表？全都有泄露风险。**KMS 让"密钥永不出加密机"**，根本上规避这个问题。

### 7.3 DSC（数据安全中心）

DSC 干 3 件事：
- **敏感数据识别**：扫 RDS / OSS，自动找出哪些字段是手机号 / 身份证 / 银行卡
- **数据脱敏**：测试环境用脱敏数据，避免开发拿到生产真实数据
- **异常访问告警**：突然有人一次拉走 10 万条手机号——告警

### 7.4 给本项目做数据安全（最务实清单）

1. RDS 创建时勾选 **存储透明加密**（开了就一直开着，无感知）
2. coffee-app 全程 HTTPS（前端 SLB 挂证书）
3. 订单表 `phone` / `address` 字段 → **KMS 加密后存**（写一个 Encryptor 工具类）
4. 备份策略：RDS 自动备份 + **跨地域备份**（异地容灾）

   > 📷 截图占位：KMS 控制台 / RDS 加密配置
   > 🔗 官方文档：[阿里云 KMS](https://www.aliyun.com/product/kms)

## Part 8 云堡垒机 — 再无"删库跑路"

> **对照前面**：你 06 章用"会话管理"登 ECS——这只解决了"不用记 SSH 密码"，**没解决"谁登过、执行了什么命令、能不能拦截 rm -rf"**。这一节用堡垒机彻底解决。

### 8.1 堡垒机解决什么

```
传统：每个运维都有 ECS 的 root 密码
              ↓
     谁登过都不知道、做了什么也没记录
              ↓
        rm -rf /* 一秒钟不可逆

堡垒机：所有运维必须先登堡垒机，再由堡垒机代登 ECS
              ↓
   ┌─────────────────────────────┐
   │  谁登 = RAM 子账号 + MFA      │
   │  登什么 = 按授权才能看到资产   │
   │  做什么 = 命令级录像 + 审计    │
   │  危险命令 = 实时拦截 + 二次审批 │
   └─────────────────────────────┘
```

### 8.2 堡垒机的核心能力

1. **统一登录入口**：所有运维必须通过堡垒机才能登 ECS / RDS / 网络设备
2. **会话审计**：每一次登录都生成一条记录——谁、何时、哪个 IP、登了哪台机器
3. **命令录像**：所有终端操作录成"视频"，可回放
4. **命令检索**：搜"谁执行过 `rm -rf /*`"——一键搜出来
5. **高危命令拦截**：配置黑名单（`drop database`、`rm -rf /*` 等），**实时阻断**
6. **运维二次审批**：业务方申请登生产数据库 → 主管在堡垒机点同意才放行

### 8.3 "再无删库跑路"是怎么做到的

3 件事一起做：
- **拦截**：危险命令直接被堡垒机当场阻断，根本执行不了
- **审批**：访问生产数据库要二级审批，单人无法独立操作
- **回溯**：即便有漏网之鱼，命令录像 1 分钟定位操作人

历史上"删库跑路"的事故无一例外都是 **运维独自掌握 root 密码 + 没人在看**——堡垒机把这两件事都堵死。

   > 📷 截图占位：堡垒机命令录像回放界面
   > 🔗 官方文档：[阿里云堡垒机](https://cn.aliyun.com/product/bastionhost)

### 8.4 给本项目接堡垒机（思路）

1. 买一台堡垒机实例（按运维人数选规格）
2. 把 3 台 ECS 导入堡垒机的"资产"
3. RAM 子账号导入堡垒机的"用户"
4. 授权：A 同学只能登 ECS-1，B 同学只能登 RDS
5. 配置高危命令黑名单
6. 关闭 ECS 的 22 端口对公网 → **唯一登录入口变成堡垒机**

## Part 9 应用架构的 5 条安全底线

不管业务多大、技术栈多花哨，下面 5 条一条都不能破：

| # | 底线 | 一句话验证 |
|---|------|----------|
| 1 | **绝不把 AK / 密码写进代码** | 用环境变量或 KMS；写了立刻轮换 |
| 2 | **绝不让生产数据库暴露公网** | RDS 白名单只放 VPC 内网 + 堡垒机 |
| 3 | **绝不让运维直接登 Root** | 全部走子账号 + 堡垒机 + ActionTrail 留痕 |
| 4 | **绝不让敏感数据明文存储** | RDS 透明加密 + 字段级 KMS 加密 |
| 5 | **绝不让公网入口裸奔** | 公网入口前面挂 WAF + DDoS + 速率限制 |

**这 5 条做到了，你的项目就过 80% 的安全审计**。剩下 20% 是流程性的——制度文档、应急预案、定期演练。

## 附录 常见问题

**Q：教学项目要不要全配齐？**

不需要。**教学版最小必要集**：
1. RAM 子账号 + ActionTrail（必做，零成本）
2. 云安全中心基础版（默认就开了，看一眼漏洞）
3. RDS 白名单收紧 + 透明加密（必做，零成本）
4. DDoS 基础防护（默认开，免费）
5. WAF（可选；要 ICP 备案 + 域名 + 几百块/月）

**Q：WAF 接入会拖慢网站吗？**

会引入约 50ms 延迟（CNAME 接入需多一跳）。对绝大多数业务可接受。透明接入延迟更低。

**Q：堡垒机贵不贵？**

按"资产数 + 运维数"计费，最低规格 3000-5000/年。教学项目用一个最低配版本就够。

**Q：等保过了就万事大吉吗？**

不是。**等保是基线，不是终点**。过等保只是说"你做到了基础保护"，真实生产里还要：
- 持续监控（云监控 + SLS）
- 定期渗透测试
- 应急演练
- 红蓝对抗

**Q：项目里发现已经把 AK 推到 GitHub 了，怎么办？**

立刻 3 步：
1. RAM 控制台 **禁用** 那对 AK
2. 生成新的 AK，更新所有用到的地方
3. 用 `git filter-branch` 或 BFG 把历史中的 AK 彻底清除（**仅删 commit 不够，git 历史还在**）
4. 如果是组织仓库，让所有协作者重新 clone

> 这是云安全中心 **AK 泄露检测** 的核心价值——它能在你被攻击前抢先告诉你。
